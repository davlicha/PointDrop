# Посібник з контриб'юції (Contributing Guidelines)

Вітаємо в команді розробки проєкту **PointDrop**! Цей документ описує єдині стандарти та правила роботи з кодом для всіх учасників команди. Дотримання цих правил допомагає уникати конфліктів, підтримувати чистоту історії коду та полегшує процес Code Review.

---

## 1. Робочий процес (Git Workflow)

Ми використовуємо спрощену версію Git Flow. У репозиторії є дві головні гілки:

* `main` — **Production-ready код**. Заборонено робити прямі коміти в цю гілку. Сюди код потрапляє виключно після релізу з гілки `develop`.
* `develop` — **Головна гілка розробки**. Це база для всіх нових фіч. Код сюди потрапляє **тільки через Pull Requests (PR)**.

### Алгоритм роботи над задачею:
1. Перейдіть у гілку `develop` та оновіть її: `git checkout develop && git pull`.
2. Створіть нову робочу гілку для вашої задачі (див. правила нижче).
3. Напишіть код, створіть коміти.
4. Відправте гілку на GitHub: `git push origin <назва_гілки>`.
5. Створіть Pull Request у гілку `develop`.
6. Дочекайтеся Code Review від іншого члена команди.
7. Після схвалення (Approve) змержіть код.

---

## 2. Правила найменування гілок (Branch Naming)

Назва гілки має чітко відображати, над чим ви працюєте. Використовуйте наступний формат: `<тип>/<опис_англійською>`.

**Типи гілок:**
* `feat/` — для розробки нового функціоналу (наприклад: `feat/auth-module`, `feat/qr-scanner`).
* `fix/` — для виправлення багів (наприклад: `fix/db-connection`, `fix/login-crash`).
* `docs/` — для роботи з документацією.
* `refactor/` — для рефакторингу коду без зміни його логіки.

**Приклад створення гілки:**
`git checkout -b feat/p2p-transfers`

---

## 3. Стандарт комітів (Conventional Commits)

Ми використовуємо стандарт **Conventional Commits** для формування історії змін. Повідомлення коміту повинно бути написане англійською мовою і мати наступну структуру:
`<type>(<scope>): <subject>`

**Допустимі типи (`<type>`):**
* `feat:` — новий функціонал (напр., додали ендпоінт).
* `fix:` — виправлення помилки/бага.
* `docs:` — зміни в документації (README, Swagger).
* `style:` — форматування коду (пробіли, крапки з комою), що не впливає на логіку.
* `refactor:` — зміна коду, яка не виправляє баг і не додає нову фічу (напр., перейменування змінних).
* `test:` — додавання або виправлення тестів.
* `chore:` — рутинні задачі (оновлення залежностей, налаштування Docker).

**Приклади правильних комітів:**
* `feat: add user registration endpoint`
* `fix(db): resolve prisma migration conflict`
* `test(transactions): add unit tests for transfer service`
* `chore: update docker-compose config`

**Заборонено комітити з повідомленнями:**
* `fixed bug`
* `update`
* `asdfasdf`

## 4. Правила створення Pull Request (PR)

Кожен PR — це запит на додавання вашого коду в загальну базу (`develop`). 

* **Опис PR:** Завжди додавайте короткий опис того, що було зроблено.
* **Зв'язок із Task Tracker:** Якщо ви використовуєте Jira/Trello, вкажіть номер задачі в описі (напр., `Resolves POIN-21`).
* **Обов'язкові перевірки (Status Checks):**
  * Переконайтеся, що проєкт локально збирається через `docker-compose up --build` без помилок.
  * Усі наявні тести (`npm run test`) мають проходити успішно.
  * Код має відповідати правилам лінтера.
* **Code Review:** Жоден PR не може бути змерджений автором. Потрібно мінімум 1 схвалення (Approve) від іншого розробника.

---

## 5. Робота з базою даних (Міграції)

Якщо ваша задача вимагає зміни структури бази даних:
1. Оновіть файл `schema.prisma`.
2. Створіть нову міграцію з описовою назвою: `npx prisma migrate dev --name <опис>`.
3. Обов'язково додайте згенеровані файли міграцій до свого коміту.
4. **Заборонено:** Змінювати таблиці напряму через `pgAdmin` або інші клієнти БД. Усі зміни — виключно через код.

Дякуємо за ваш внесок у розвиток PointDrop!
